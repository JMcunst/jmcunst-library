# 구조 설계 (Structural Design)

## 정의

> 특정 문제를 소프트웨어로 해결하기로 결정했을 때, 그 문제를 **어떤 구조로 풀 것인가**를 정하는 일

단순히 기능을 나누는 수준이 아니라:
- 어떤 요소들이 있고
- 이 요소들이 어떻게 관계를 맺고
- 어떤 특성을 우선적으로 만족시킬 것인가

를 **전체 관점**에서 결정하는 것이 구조 설계다.

---

## 구조 설계의 두 축

구조 설계는 **반드시 두 영역을 분리해서** 생각해야 한다:

| 영역 | 핵심 질문 |
|------|----------|
| **도메인 (Domain)** | 무엇을 해결해야 하는가? |
| **아키텍처 특성 (Architecture Characteristics)** | 어떻게 잘 해결할 것인가? |

---

## 도메인 (Domain)

> "무엇을 해결해야 하는가"

소프트웨어가 풀어야 할 **문제의 본질**

### 도메인이 다루는 것
- 비즈니스 요구
- 기능 요구사항
- 데이터 자체의 의미

### 예시
- 주문을 생성해야 한다
- 결제를 처리해야 한다
- 재고를 차감해야 한다

---

## 아키텍처 특성 (Architecture Characteristics)

> "어떻게 잘 해결할 것인가"

도메인과 직접적인 기능은 아니지만, **시스템의 성공을 좌우하는 비기능적 특성들**

### 대표적인 아키텍처 특성
- 감시성 (Observability)
- 성능 (Performance)
- 보안 (Security)
- 확장성 (Scalability)
- 모듈성 (Modularity)
- 데이터 관리 특성
- 적합성

### 다양한 이름으로 불림
- 아키텍처 특성
- 비기능 요구사항
- 시스템 품질 속성
- 공통 요구 사항

(용어 통일 강요 ❌)

---

## 아키텍처 특성이 중요한 이유

아키텍처 특성은 **프로젝트 전 생애주기**에 영향을 준다:
- 초기 릴리스부터
- 장기 운영·유지보수 단계까지

| 특성 | 영향 |
|------|------|
| 확장성, 성능 | 시장 성공 가능성에 직결 (트래픽 폭증, 이벤트 대응) |
| 모듈성 | 유지보수성, 변경 비용 감소, 진화 가능성 |

### 핵심 통찰

```
기능은 "작동 여부"를 결정하고
아키텍처 특성은 "살아남을 수 있는지"를 결정한다
```

---

## 구조 설계의 현실적 관점

### 정적인 소프트웨어는 거의 없다

- 기능이 계속 추가되고
- 통합 지점이 늘어나고
- 기술 정책, 외부 환경 때문에
- **도메인과 무관하게 구조 변경이 발생한다**

### 구조 설계의 목적

> 변화를 전제로 한 보호 메커니즘을 만드는 것

- 변화가 와도 전체가 무너지지 않도록
- 도메인과 아키텍처 특성이 서로 과도하게 얽히지 않게

### 비유

```
단위 테스트가 코드 변경을 보호하듯
구조 설계는 아키텍처 특성의 변경 주기를 보호한다
```
