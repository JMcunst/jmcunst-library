# 비트 부패 (Bit Rot)

## 정의

소프트웨어가 **아무것도 변경하지 않았는데도** 시간이 지나면서 점점 동작하지 않게 되는 현상

> 코드는 그대로인데, 세상이 변했다

## 왜 발생하는가?

소프트웨어는 진공 상태에서 존재하지 않는다. 주변 환경이 계속 변한다:

| 변화 요인 | 예시 |
|----------|------|
| 의존성 업데이트 | 라이브러리 버전 변경, deprecated API |
| 플랫폼 변화 | OS 업데이트, 런타임 버전 변경 |
| 인프라 변화 | 클라우드 서비스 정책 변경, 인증 방식 변경 |
| 보안 요구사항 | 새로운 취약점 발견, 암호화 표준 변경 |
| 외부 API 변경 | 서드파티 API 스펙 변경, 서비스 종료 |

## 왜 중요한가?

- 방치된 시스템은 **점점 더 고치기 어려워진다**
- 나중에 한꺼번에 업그레이드하면 → 대규모 마이그레이션 지옥
- 보안 취약점 누적

## 비트 부패의 증상

- "예전엔 됐는데 지금은 안 돼요"
- 오래된 서비스 재배포하면 실패
- 의존성 업데이트하면 빌드가 깨짐
- 새 팀원이 로컬 환경 세팅 실패

## 대응 방법

### 1. 지속적인 작은 변경
- 정기적인 의존성 업데이트
- 작은 단위로 자주 배포

### 2. 자동화된 검증
- CI/CD 파이프라인
- 피트니스 함수로 아키텍처 특성 모니터링

### 3. 의존성 관리
- Dependabot, Renovate 등 자동화 도구
- 버전 고정 vs 범위 지정 전략

### 4. 문서화
- 빌드/배포 절차 문서화
- 의존성 결정 이유 기록 (ADR)

## 실무 한 줄

> 시스템을 오래 방치할수록, 나중에 치러야 할 비용은 기하급수적으로 늘어난다
