# USE와 RED의 상호보완 (Complementary USE+RED Monitoring)

## 정의

USE(인프라 자원 관측)와 RED(애플리케이션 품질 관측)를 함께 사용해, 증상 탐지와 원인 추적을 동시에 빠르게 수행하는 운영 방식.

## 왜 중요한가?

- RED만 보면 "서비스가 나쁘다"는 사실은 알지만 인프라 병목 위치가 모호할 수 있다.
- USE만 보면 자원 병목은 찾지만 사용자 영향도를 놓치기 쉽다.
- 두 프레임워크 결합 시 탐지 정확도와 대응 속도가 모두 향상된다.

## 책 내용 핵심 정리

- 두 방법론은 상호보완적 관계다.
- USE는 인프라 컴포넌트에 집중하고, RED는 애플리케이션 엔드 유저 경험 중심으로 본다.

## 실무 포인트

### 1. 알람 체계 이원화
- 1차 알람: RED (사용자 영향 탐지)
- 2차 진단: USE (원인 자원 식별)

### 2. 대시보드 연결
- RED 패널에서 바로 노드/파드 USE 지표로 드릴다운한다.
- 같은 시간축으로 배포 이벤트를 함께 본다.

### 3. 사고 대응 표준화
- 장애 대응 런북에 `RED 확인 -> USE 진단 -> 복구` 순서를 명시한다.
- 회고 시 두 지표를 함께 검토해 재발 방지 액션을 도출한다.

## 바로 적용 가능한 체크리스트

- [ ] RED 알람 발생 시 USE 진단 절차가 문서화되어 있다.
- [ ] 서비스 대시보드에서 인프라 지표로 바로 이동 가능하다.
- [ ] 런북에 공통 진단 순서(RED->USE)가 정의되어 있다.
- [ ] 장애 회고에 사용자 영향 지표와 자원 지표를 함께 기록한다.

## 실무 한 줄

RED로 문제를 발견하고 USE로 원인을 좁히는 흐름이, 쿠버네티스 운영에서 가장 실용적인 모니터링 패턴이다.

## 관련 개념

- Incident triage
- Correlated dashboards
- Root cause isolation
- Platform observability
