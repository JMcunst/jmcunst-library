# 개발 단계 KPI (Development Stage KPI)

## 정의

개발자가 코드 변경을 빠르게 반복하고 클러스터에서 검증/디버깅할 수 있도록, 개발 단계의 리드타임과 체감 생산성을 측정하는 방법.

## 왜 중요한가?

- 배포 파이프라인이 길면 개발 이터레이션 속도가 급격히 저하된다.
- 디버깅 대기 시간이 길수록 기능 완성까지 시간이 늘어난다.
- 팀이 느끼는 생산성과 시스템 지표가 다를 수 있어 둘 다 필요하다.

## 책 내용 핵심 정리

- 두 번째 단계는 개발이며, 목표는 신속한 이터레이션과 디버깅이다.
- KPI 측정은 쉽지 않지만, PR 또는 코드 수정 후 클러스터 실행까지 걸린 시간과 개발자 체감 생산성을 활용할 수 있다.

## 실무 포인트

### 1. 리드타임 지표를 고정한다
- 코드 푸시 -> 이미지 빌드 완료
- 빌드 완료 -> 개발 클러스터 반영
- 반영 -> 검증 성공

### 2. 디버깅 루프를 단축한다
- 개발 환경 로그/메트릭 접근 권한을 기본 제공한다.
- 원클릭 재배포/롤백 같은 빠른 실험 경로를 만든다.

### 3. 정성 지표를 함께 본다
- 개발자 설문(속도, 안정성, 신뢰도)
- 주간 생산성 체감 점수
- 반복 장애에 대한 불편 지점 수집

## 바로 적용 가능한 체크리스트

- [ ] 코드 변경 후 개발 클러스터 반영 시간을 측정한다.
- [ ] 디버깅에 필요한 관측 도구 접근을 표준화했다.
- [ ] 개발자 체감 생산성 설문을 정기 수집한다.
- [ ] 상위 병목 3개를 분기마다 제거한다.

## 실무 한 줄

개발 단계 KPI의 핵심은 "배포 속도"가 아니라 "문제 확인까지 걸리는 시간"이다.

## 관련 개념

- Lead Time for Change
- Inner Loop Optimization
- Developer Productivity Survey
- Fast Feedback
